Conclusion for Route A:
- The heap-handle consumer for .dG3_DXE is recovered from current bytes.
- The seven dwords at 0xFFF8D0EC are heap handles, not P-state records.
- The function pointer 0xFFF8D11E -> 0xFFF7371A is in a nearby static function-pointer array, not reached by a direct E8 call.
- Function 0xFFF7371A is a static firmware entry point, but the P-state structure it consumes through arg_0C is runtime/heap-selected.
- arg_0C is best classified as a CPU-family/runtime dispatch record or heap-selected table context, not a byte-ready static P4 table row.
- No P4-only static editable table target is proven by this artifact.
