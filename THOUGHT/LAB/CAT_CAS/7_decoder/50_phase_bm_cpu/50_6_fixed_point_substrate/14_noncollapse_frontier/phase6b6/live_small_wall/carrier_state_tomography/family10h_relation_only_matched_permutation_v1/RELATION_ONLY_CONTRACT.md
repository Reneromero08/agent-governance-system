## Relation Grammar

The public relation labels select public permutation tables before the measured
loop. They do not select relation-specific hot-loop code.

* `relation_r0`: `B_index = (A_index + 1) mod 4096`
* `relation_r1`: `B_index = (A_index - 1) mod 4096`

The source walks A through one public traversal, starting at a balanced cyclic
origin, and maps B through the selected permutation table.

The receiver walks the selected query relation with the same origin law.

Frozen cyclic origins:

* `0`
* `1024`
* `2048`
* `3072`

Relation cells:

* prepare r0, query r0
* prepare r0, query r1
* prepare r1, query r0
* prepare r1, query r1

Primary endpoint:

`dirty_probe_response`

Primary relation coordinate:

`R_match = 0.5 * ((Y_r0_r0 + Y_r1_r1) - (Y_r0_r1 + Y_r1_r0))`

Positive or negative sign may carry the relation-match coordinate. Sign alone
does not establish a physical mechanism.

## Runtime Semantics

Every acquired tuple must use a fresh carrier state.

Required source preparation:

1. Allocate owned A/B carrier lanes and matched sham lanes.
2. Prefault all lanes before source fork.
3. Pin source to the frozen source CPU.
4. Select relation permutation table before the loop.
5. Iterate A indices through the frozen traversal and cyclic origin.
6. Map B through the selected relation table.
7. Apply the source-order factor with matched counts.
8. Terminate the source before receiver query.
9. Emit a source-death receipt.

Required receiver query:

1. Pin receiver to the frozen receiver CPU.
2. Enforce the frozen delay.
3. Select query relation before PMU acquisition.
4. Acquire the grouped PMU event set.
5. Execute the same compiled query loop for both relation labels.
6. Emit `change_to_dirty`, `dirty_probe_response`, `cpu_cycles`, and
   `duration_ns`.

The current source implements the relation-table preparation/query primitives
and the fail-closed runtime authority gate. The generated build-readiness
receipt determines whether the local compile and runtime self-test gates have
actually passed.

## Executable Controls

Physical schedule rows include these controls:

* `relation_sham`: matched paired traversal with no stable prepare-query
  relation correspondence.
* `route_pressure_sham`: matched interleaving, instruction count, and memory
  pressure with relation-neutral pairing.
* `independent_marginal_replay`: A and B marginal work without preserving pair
  correspondence.
* `distance_control`: pair-distance histogram preserved while correspondence is
  intentionally mismatched.

`label_scramble` is not a physical schedule row. It is an offline adjudication
regression over retained raw evidence.
