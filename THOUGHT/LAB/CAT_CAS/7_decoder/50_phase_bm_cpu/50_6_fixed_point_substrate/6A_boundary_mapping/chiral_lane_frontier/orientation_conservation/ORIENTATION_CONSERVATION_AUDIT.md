## 0. Purpose

Before building Track A's `chiral_dual_lane.rs`, verify that the public schedule
emitted by the oracle generator is invariant under the private fold `d <-> N-d`.
A schedule that changes under the fold would smuggle orientation information into
the runtime, invalidating any public crossing claim before physical execution begins.

## 1. The Odd-Lane Source for Track A

**Named odd-lane source: intermediate integer register values during the phase walk
accumulation, coupled through the PDN common-mode rejection differential.**

**How it works:**
- Two sender cores run the phase walk simultaneously: one with `candidate_0 = a`,
  one with `candidate_1 = N-a`.
- At step `j`, the intermediate register holds `phi_j = candidate * K_j mod N`,
  where `K_j` is the cumulative sum of `k_i` values up to step `j`.
- For `candidate_0 = a`: `phi_j = a * K_j mod N`.
- For `candidate_1 = N-a`: `phi_j = (N-a) * K_j = -a * K_j mod N`.
- The per-step bit patterns of these intermediate values differ (different Hamming
  weights, different carry-chain activity in the ALU).
- The PDN current draw responds to the integer datapath activity, NOT just the
  final cosine output.
- The differential between the two cores' PDN signatures measures the asymmetry
  between `a * K_j` and `(N-a) * K_j` intermediate values.

**Orientation coupling assessment:**
- The differential separates candidate_0 from candidate_1 (the values `a` vs `N-a`).
- This is CANDIDATE-VALUE coupling, not orientation coupling, as established by the
  boundary analysis.
- Whether the differential carries the hidden orientation `b_orient = 1[d < N/2]`
  depends on whether the PDN differential correlates with the true `d` beyond what
  is predictable from the public candidate values.
- Track A's data will ADJUDICATE this, not assume it.

**Where the odd representation enters:**
- Physical CPU execution geometry (PDN current draw from integer multiply stages).
- NOT from the public data. NOT from hidden `d`. NOT from smuggled labels.
- The odd representation is generated by the hardware's response to different
  integer operands `a * k_i` vs `(N-a) * k_i`.

## 3. Candidate Label Verification

For every instance:
- `candidate_0 = a = min(d, N-d)` -- always the smaller value (fold-invariant).
- `candidate_1 = N-a = max(d, N-d)` -- always the larger value (fold-invariant).
- The labels are `candidate_0` and `candidate_1`; never `true`, `false`, `d`, `N-d`,
  `left`, `right`, `orientation`, or any orientation-bearing name.

**Blinding confirmation:** The runtime receives only the values `candidate_0` and
`candidate_1`. It does not know which equals the hidden `d`. The offline scorer
maps these to true/false after the run completes.
